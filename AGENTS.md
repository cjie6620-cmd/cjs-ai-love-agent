# AGENTS.md

## 1. 文档目标

本文件用于定义本项目的统一开发规范、Agent 工程约束、代码注释要求、技术栈边界、配置与安全规则。

本文件适用于以下参与者：

- 后端开发工程师
- AI / Agent 开发工程师
- 前端开发工程师
- 运维与测试工程师
- 参与本仓库协作的 AI Coding Agent


本项目在吸收这些开源项目风格的基础上，结合“AI 恋爱智能体”业务特性，形成适合企业落地的项目规范。

---

## 2. 项目定位

本项目是一个面向情感陪伴、恋爱沟通、风格复刻、长期记忆与安全治理的企业级 AI Agent 系统。

项目目标不是实现一个简单聊天 Demo，而是建设一个具备以下特点的正式工程：

- 可持续迭代
- 可观测和可评估
- 可控制和可审计
- 可商业化和可部署

## 4. 目录与模块约定

推荐项目结构如下：

```text
app/
  api/          # API 路由层
  schemas/      # Pydantic 输入输出模型
  domain/       # 领域模型与业务规则
  services/     # 业务服务
  agents/       # Agent 工作流、节点、工具
  prompts/      # Prompt 模板与版本
  memory/       # 记忆管理与摘要逻辑
  rag/          # 检索、索引、重排能力
  safety/       # 安全治理与风控规则
  tasks/        # Celery 任务
  infra/        # 数据库、缓存、对象存储、模型客户端
tests/
docs/
```

约定如下：

- 路由层只做协议处理，不承载复杂业务逻辑。
- 业务逻辑优先进入 `services/` 或 `domain/`。
- Agent 编排、节点流转、工具注册统一放入 `agents/`。
- Prompt 不允许散落在任意业务文件中，必须统一管理。
- 安全治理逻辑不允许只写在前端或调用方，必须后端兜底。

## 5. 代码规范

### 5.1 通用要求

- 所有代码必须保持可读性优先。
- 优先选择最小正确改动，不做无必要抽象。
- 命名必须语义清晰，避免缩写泛滥。
- 新增功能必须明确输入、输出、失败路径。
- 严格进行空值判断和异常处理。

### 5.2 Python 代码要求

- 使用类型注解。
- 数据输入输出统一使用 `Pydantic` 模型。
- 数据库访问统一通过仓储或服务层，不允许在路由层直接拼接 SQL。
- 异步接口优先使用 `async` 风格，但不要无意义全异步化。
- 与外部模型、向量库、对象存储交互时必须增加日志和异常保护。

### 5.3 Agent 代码要求

- Agent 节点职责单一，便于追踪和回放。
- 工作流必须显式定义状态流转，不允许隐式串联。
- Prompt、工具、记忆、风格、安全必须分层。
- 所有工具调用都要定义结构化输入输出。
- 关键节点应支持 trace 和调试日志。

## 6. 注释规范

### 6.1 总体要求

本项目所有新增代码注释必须使用中文，要求通俗、准确、可读。

### 6.2 注释原则

- 解释“为什么这样做”，而不只是解释“代码正在做什么”。
- 复杂业务规则、边界条件、风控逻辑、Prompt 设计原因必须写注释。
- 不要添加无意义注释，例如“给变量赋值”。
- 函数、类、复杂流程节点建议增加中文文档注释。

### 6.3 推荐示例

```python
def build_style_prompt(profile: StyleProfile, samples: list[str]) -> str:
    """根据角色画像和样本表达构造风格注入提示词。"""
    # 这里不直接拼接全部聊天记录，只拼接经过筛选的高质量样本，避免上下文污染和风格漂移。
    ...
```

## 7. Prompt 与 Skill 规范

### 7.1 Prompt 规范

- Prompt 必须模块化管理。
- Prompt 必须支持版本号。
- Prompt 变更需要记录变更原因。
- 安全 Prompt 不允许被业务 Prompt 覆盖。

### 7.2 Skill 规范

每个 Skill 至少包含：

- 使用场景说明
- 输入结构
- 输出结构
- Prompt 模板
- 安全规则
- 示例用例

### 7.3 禁止事项

- 禁止将所有逻辑堆入一个超长系统 Prompt。
- 禁止在多个文件中复制粘贴相同 Prompt。
- 禁止未评估直接上线新的高风险风格 Prompt。

---

## 8. RAG 与记忆规范

### 8.1 RAG 规范

- 知识检索、长期记忆检索、风格样本检索必须分库或分集合管理。
- 检索必须优先支持元数据过滤。
- 重要场景建议使用混合检索与重排。

### 8.2 记忆规范

- 短期上下文和长期记忆必须分层。
- 不允许把全部历史消息直接拼入上下文。
- 长期记忆写入前必须经过摘要或结构化提炼。
- 敏感记忆要支持删除和脱敏。

---

## 9. 安全治理规范

### 9.1 必须具备的能力

- 输入审核
- 输出审核
- 高风险情绪识别
- 依赖强度监测
- 危机场景兜底策略
- 审计日志

### 9.2 高风险场景处理原则

- 安全优先于风格一致性。
- 风险用户优先进入降温和边界控制模式。
- 不允许模型输出鼓励伤害、过度依附、现实替代型内容。

---

## 10. 配置与密钥管理规范

### 10.1 配置管理要求

- 配置项写入文档和示例文件。
- 真实密钥不得提交到 Git 仓库。
- 所有敏感配置必须支持环境变量覆盖。

### 10.2 密钥管理要求

- 数据库密码、对象存储密码、第三方 API Key 一律视为敏感信息。
- 本地使用 `application-local.yml`。
- 测试和生产环境使用 Secret 管理系统。
- 已经暴露过的密钥必须立即轮换。

## 11. 测试与质量要求

### 11.1 测试要求

- 核心业务服务必须有单元测试。
- Agent 工作流至少覆盖关键路径测试。
- RAG 检索逻辑应有最小可用评估集。
- 高风险安全规则必须有回归测试。

### 11.2 质量工具建议

- `pytest`
- `ruff`
- `mypy` 或 `pyright`
- `coverage`

### 11.3 合入前检查

- 代码格式通过
- 静态检查通过
- 关键测试通过
- 无新增明文密钥
- 文档同步更新

## 12. Git 与文档协作规范

### 12.1 Git 规范

- 不提交真实密码、Token、证书。
- 不提交本地运行缓存和私有配置。
- 配置变更需同步更新文档或示例文件。

### 12.2 文档规范

- 架构变更必须同步更新 `PROJECT_PLAN_PYTHON_ENTERPRISE.md`。

- 新增模块建议补充 README 或模块设计说明。

---

## 13. 本项目的开发底线

- 不为追求“更像真人”而突破安全边界。
- 不为追求“更智能”而牺牲可控性和可审计性。
- 不为追求“快速上线”而忽略密钥泄露和隐私风险。
- 不引入与当前阶段不匹配的复杂基础设施。

本项目最终目标是构建一个企业可落地、用户可持续使用、团队可持续维护的 AI 恋爱智能体系统。
