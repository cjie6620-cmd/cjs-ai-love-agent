# contracts 模块说明

## 1. 模块定位

`contracts` 是共享契约层，负责定义跨模块传输的数据结构、请求对象、响应对象和工作流共享模型。

它解决的是“模块之间如何稳定通信”的问题。

## 2. 目录结构

- `chat.py`
  作用：聊天相关契约。
  功能：定义聊天请求、响应、追踪信息、顾问结果、会话消息和访客额度字段等模型。
- `auth.py`
  作用：认证相关契约。
  功能：定义登录/注册入参、Token 响应、当前用户响应和刷新令牌请求。
- `common.py`
  作用：通用契约。
  功能：定义健康检查、通用错误响应等基础结构。
- `rag.py`
  作用：RAG 相关契约。
  功能：定义索引请求、搜索请求、检索结果等对象。
- `__init__.py`
  作用：模块导出入口。
  功能：统一暴露常用契约。

## 3. 对外职责

- 统一 API、Service、Workflow 之间的数据模型。
- 统一前后端认证、聊天、RAG 和健康检查的字段形态。
- 统一字段命名、类型约束和序列化规则。
- 减少“字典拼装”和隐式字段漂移。

## 4. 依赖边界

- 允许被上层各模块复用。
- 不应依赖业务服务、数据库实现或第三方调用逻辑。
- 契约层要保持轻量、稳定、可复用。

## 5. 企业规范做法

- 所有跨模块传输对象优先沉淀到契约层。
- 禁止在多个模块里维护重复结构。
- 字段修改要评估兼容性，避免直接破坏调用方。
- 契约命名要业务化，避免 `Data`、`Info2` 这类模糊名称。
- 重要字段要带说明，便于接口和链路追踪使用。
- `ChatRequest.user_id` 只能作为后端内部注入字段，前端请求不应依赖它。

## 6. 维护要求

- 新增共享对象时同步更新本 README。
- 修改字段时补充测试，并检查 API、前端、工作流是否受影响。
