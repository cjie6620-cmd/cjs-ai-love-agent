# llm 模块说明

## 1. 模块定位

`llm` 是大模型接入层，负责统一封装模型调用、Provider 路由、重试、工具注册和结构化输出能力。

它解决的是“业务层如何稳定调用不同模型能力”的问题。

## 2. 目录结构

- `client.py`
  作用：LLM Facade 入口。
  功能：向上层暴露统一的大模型调用接口。
- `factory.py`
  作用：Provider 工厂。
  功能：根据配置创建具体模型提供方实例。
- `retry.py`
  作用：重试策略模块。
  功能：处理模型调用中的可恢复失败。
- `core/`
  作用：基础类型目录。
  功能：定义 LLM 内部共享类型和抽象。
- `providers/`
  作用：模型提供方实现目录。
  功能：封装 OpenAI、xAI Router 等具体实现。
- `tools/`
  作用：模型工具体系。
  功能：统一管理 Tavily、MCP 等工具调用能力。
- `tests/`
  作用：模块测试目录。
  功能：验证结构化输出、分词策略、工具路由等行为。
- `__init__.py`
  作用：模块公共出口。
  功能：导出常用客户端或接口。

## 3. 对外职责

- 统一调用不同模型 Provider。
- 提供结构化输出和流式输出能力。
- 提供模型工具编排与调用跟踪。

## 4. 依赖边界

- 可以依赖：`core`、`prompt`、`mcp`、`observability`
- 不应直接依赖业务路由和页面逻辑。
- 业务模块统一通过 LLM 客户端或工厂访问模型。

## 5. 企业规范做法

- Provider 与业务隔离，新增模型只改接入层，不改业务层。
- 工具注册与模型调用分层，避免 Provider 内部写死业务工具。
- 统一超时、重试、追踪、结构化输出规范。
- 任何第三方 SDK 差异都应收敛在 Provider 或 Client 适配层。
- 输入输出要做归一化，避免把各家模型响应格式泄漏给上层。

## 6. 维护要求

- 新增 Provider、Tool Client 或重试策略时同步更新本 README。
- 改动模型调用协议时补测试，特别是流式和结构化输出路径。
