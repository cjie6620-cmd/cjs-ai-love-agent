# observability 模块说明

## 1. 模块定位

`observability` 是可观测性模块，负责日志、链路追踪、请求关联标识和 LangSmith 集成。

它解决的是“系统出了问题时，团队如何快速看见、定位、回放”的问题。

## 2. 目录结构

- `ai_logger.py`
  作用：结构化日志系统。
  功能：统一日志格式、关联 ID 注入和中间件注册。
- `langsmith_service.py`
  作用：LangSmith 服务封装。
  功能：负责 tracing、脱敏、Prompt 拉取与客户端包装。
- `tests/`
  作用：观测模块测试。
  功能：验证 LangSmith 集成与日志相关能力。
- `__init__.py`
  作用：统一导出入口。
  功能：暴露日志和 tracing 常用方法。

## 3. 对外职责

- 提供结构化日志能力。
- 提供调用链追踪能力。
- 提供敏感信息脱敏和请求关联标识。

## 4. 依赖边界

- 可以被 API、LLM、Agent 等模块复用。
- 不承载业务流程，不直接操作业务数据。
- 观测逻辑必须尽量少侵入主业务代码。

## 5. 企业规范做法

- 日志、trace、指标职责分清，不混在业务代码里硬写。
- 敏感字段上报前必须脱敏。
- trace metadata 要可搜索、可过滤、可定位。
- 中间件和日志格式在模块内统一收敛，不要每层各写一套。

## 6. 维护要求

- 新增链路追踪字段或日志格式时同步更新本 README。
- 涉及隐私数据时先定义脱敏规则，再接 tracing。
