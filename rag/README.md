# rag 模块说明

## 1. 模块定位

`rag` 是知识检索增强模块，负责文档加载、解析、切分、Embedding、向量存储、检索与融合。

它解决的是“模型如何基于项目知识库回答问题”的问题。

## 2. 目录结构

- `data_loader.py`
  作用：知识文件加载与清洗。
  功能：读取知识库目录中的原始文件内容。
- `pipeline.py`
  作用：离线索引管道。
  功能：串联解析、切分、向量化、写库流程。
- `rag_service.py`
  作用：RAG 服务入口。
  功能：为 API 和 Agent 提供索引、重建、搜索能力。
- `retriever.py`
  作用：知识检索器。
  功能：把查询转成向量并检索知识结果。
- `schemas.py`
  作用：检索模型定义。
  功能：定义请求参数和检索结果结构。
- `storage.py`
  作用：对象存储适配。
  功能：封装 MinIO 文件上传下载能力。
- `chunking/`
  作用：切分策略目录。
  功能：维护文本分块抽象与具体策略。
- `embeddings/`
  作用：向量化目录。
  功能：封装 Embedding 生成逻辑。
- `fusion/`
  作用：融合策略目录。
  功能：实现多路召回结果融合，如 RRF。
- `ingestion/`
  作用：文档摄取目录。
  功能：定义解析基类、解析器与注册表。
- `vector_store/`
  作用：向量存储目录。
  功能：定义 pgvector 模型、基座与访问客户端。
- `tests/`
  作用：RAG 模块测试。
  功能：验证切分、检索、索引行为。

## 3. 对外职责

- 提供知识索引能力。
- 提供知识检索能力。
- 提供文件存储与向量存储协同能力。

## 4. 依赖边界

- 可以依赖：`core`、`llm`、`contracts`
- 供 `api` 和 `agents` 调用。
- 不直接处理路由层逻辑和页面交互。

## 5. 企业规范做法

- 摄取、切分、向量化、存储、检索分层拆开，避免大而全单文件。
- 新文档格式接入优先扩展 `ingestion/`，不要硬改主流程。
- 新切分策略优先扩展 `chunking/`，保持策略可替换。
- 检索结果要有统一结构，便于 Agent 注入和追踪。
- 重建索引要具备幂等意识，避免重复数据污染。

## 6. 维护要求

- 新增解析器、切分器、向量表或检索策略时同步更新本 README。
- 影响索引链路的改动必须补测试。
