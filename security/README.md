# security 模块说明

## 1. 模块定位

`security` 是安全治理模块，负责 API Key 认证、JWT 登录、匿名访客额度、限流、输入输出安全规则和缓存相关安全能力。

它解决的是“系统如何在可用的同时保持边界、安全和稳定”的问题。

## 2. 目录结构

- `api_key.py`
  作用：API Key 鉴权中间件。
  功能：校验请求头中的 API Key。
- `auth.py`
  作用：账号密码登录和 Token 服务。
  功能：提供密码哈希、JWT access token、refresh token 哈希和当前用户解析能力。
- `guardrails.py`
  作用：安全护栏模块。
  功能：识别高、中、低风险输入并清洗模型输出内容。
- `guest_quota.py`
  作用：匿名访客试用额度模块。
  功能：基于 Redis 记录 `guest_quota:{anonymous_id}:{date}`，控制未登录每日发送次数。
- `rate_limit.py`
  作用：限流与 Redis 能力模块。
  功能：封装令牌桶限流、通用 KV、计数器、缓存和会话上下文能力。
- `tests/`
  作用：安全模块测试。
  功能：验证风险识别、输出清洗和限流逻辑。
- `__init__.py`
  作用：统一导出入口。
  功能：暴露安全治理相关对象。

## 3. 对外职责

- 提供接口认证能力。
- 提供 JWT 登录态解析和 refresh token 管理能力。
- 提供匿名访客每日试用次数控制能力。
- 提供请求限流能力。
- 提供输入输出安全检查能力。

## 4. 依赖边界

- 可以被 API、Agent 等上层模块复用。
- 不应依赖页面层和业务流程编排实现。
- 安全规则统一收敛在本模块，其他模块不散写安全判断。

## 5. 企业规范做法

- 鉴权、限流、内容安全分层管理。
- 登录态使用后端解析出的 `users.id`，业务接口不得信任前端自传 `user_id`。
- 匿名访客额度 Redis 不可用时默认要求登录，避免免费额度失控。
- 高风险规则必须可追踪、可审计、可测试。
- 缓存能力和安全能力要明确边界，避免职责混乱。
- 安全降级策略要可控，不能悄悄放大风险。

## 6. 维护要求

- 新增安全规则或限流策略时同步更新本 README。
- 涉及高风险逻辑修改时必须补回归测试。
